# Mapperly Mappers

CLAUDE.md default `object_mapping_library: Mapperly`. Skill emits a Mapperly-annotated interface; Mapperly source-generator turns it into a concrete class at build time. AppService injects the interface.

`System.Text.Json`, reflection-based mappers, and `AutoMapper` are not alternatives. Planner's holistic pass fails the plan if `AutoMapper.Profile` is planned while CLAUDE.md says Mapperly.

## Package

Requires `Riok.Mapperly` in `<Ns>.Application` project. If scaffolding check reports package missing, final report lists it as prerequisite; skill does not modify `.csproj` files.

## Interface file

- Path: `<Ns>.Application/<Feature>/<Feature>Mapper.cs`. Namespace `<Ns>.<Feature>`.
- File header XML doc with `<see href="<feature wiki URL>"/>`.
- `using Riok.Mapperly.Abstractions;`.

```csharp
[Mapper]
public partial interface I<Feature>Mapper
{
    <Entity>Dto MapToOutput(<Entity> entity);
    IEnumerable<<Entity>Dto> MapToOutputList(IEnumerable<<Entity>> entities);
    // ... one pair per entity whose output is emitted
}
```

Mapperly generates implementation class `<Feature>Mapper` at build time.

## Property mapping

- Auto-mapped — Mapperly matches source-to-target by name and type.
- Deviations (name mismatch, type coercion) → `[MapProperty(...)]` on interface method. Planner flags need based on FS notes; synthesizer emits attribute.
- Nested DTOs auto-map if both types have matching mapping method in same `[Mapper]`.
- Collections (`IEnumerable<T>`, `List<T>`, `IReadOnlyCollection<T>`) auto-map when element types have mappings.

## Enum mapping

Enums with matching member names auto-map. Global `JsonStringEnumConverter` (camelCase) handles HTTP-boundary serialization; Mapperly is unaffected — maps enum-to-enum.

## Null handling

Mapperly default is non-throwing: mapping `null` source to reference-type target produces `null`. Do not override unless FS calls for throwing on null.

## DI registration

`<Feature>ApplicationModule.ConfigureServices`:

```csharp
context.Services.AddScoped<I<Feature>Mapper, <Feature>Mapper>();
```

(Class `<Feature>Mapper` generated by Mapperly. ABP's conventional registration also picks it up via `[Mapper]` — explicit registration documented for test-override clarity.)

## Usage in AppService

Constructor injects `I<Feature>Mapper _mapper`. Methods call:

- `_mapper.MapToOutput(entity)` — single.
- `_mapper.MapToOutputList(entities).ToList()` — collection (Mapperly returns `IEnumerable<T>`).

No manual property assignment. No `new <Entity>Dto { Name = entity.Name, ... }` in AppServices — synthesizer refuses.

## Input DTO → entity mapping

By default, **no input-to-entity mapping method is emitted**. Aggregates are constructed via `<Entity>.Create(...)` (Builder pattern), not by mapping a DTO to an entity. AppService unpacks DTO and calls factory:

```csharp
var entity = <Entity>.Create(input.Name, input.Amount);  // No tenantId — ABP handles (CRC-D1)
```

Mapperly is used exclusively for entity → DTO projection on the way out. Input mapping would bypass Builder's invariant checks.

## Partial updates

For `Update<Entity>Dto`, AppService loads aggregate and calls domain methods — does NOT project DTO-to-entity via Mapperly. Mapping `Update*Dto` fields onto entity via reflection would mutate state without routing through domain methods, violating aggregate invariants.

If FS explicitly says "mass assignment allowed" (rare; only CRUD-heavy admin tables), Entity page must declare so and planner may emit mapping method with `[MapperIgnoreSource]` on audit fields. Default: no input mapping.

## Anti-patterns the synthesizer refuses

`AutoMapper.Profile` subclasses · `IMapper` (AutoMapper generic interface) as ctor dep · reflection-based mappers (`object.MapTo<T>()` extension helpers) · manual property-by-property DTO population in AppServices · Mapperly methods returning the source type unchanged (skipped — synthesizer doesn't emit).
