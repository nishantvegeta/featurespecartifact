**Node type:** Command
**Name:** ToggleChecklistItemStatus
**Actor:** [BankAdmin](http://localhost:8080/root/trade-finance/-/wikis/actors/BankAdmin)
**Target aggregate:** [ChecklistItem](http://localhost:8080/root/trade-finance/-/wikis/entities/ChecklistItem)
**Module:** BankSettings
**Sub-module:** ChecklistManagement
**Purpose:** Switch a checklist item's status between `Active` and `Inactive`. If the current status is `Active`, the command sets it to `Inactive`; if it is `Inactive`, the command sets it to `Active`. Optimistic concurrency is enforced via `ConcurrencyStamp` to prevent stale-read conflicts when two admins operate on the same item simultaneously. Sequence number and description are not affected.
**Audience:** Private
**HTTP route:** POST /api/private/app/bank-settings/checklist-items/{id}/toggle-status

**Input DTO:** `ToggleChecklistItemStatusInput`

| Name | Type | Required | Validation | Notes |
|---|---|---|---|---|
| `Id` | `Guid` | yes | NotEmpty | Route-bound from URL path parameter |
| `ConcurrencyStamp` | `string` | yes | NotEmpty | Token from the most recent read of this item; enforces optimistic concurrency. See [OptimisticConcurrencyStrategy](http://localhost:8080/root/trade-finance/-/wikis/decisions/OptimisticConcurrencyStrategy) |

**Input DTO base:** plain (no audit inheritance)
**Validation:** `ToggleChecklistItemStatusInputValidator` (FluentValidation)
**Output DTO:** `ChecklistItemDto` (`FullAuditedEntityDto<Guid>`) — mirrors aggregate's auditing level; caller can read the new `Status` from the response
**Authorization:** `TradeFinancePermissions.BankSettings.ChecklistItems.ToggleStatus`

**Preconditions:**
- Caller holds permission `TradeFinancePermissions.BankSettings.ChecklistItems.ToggleStatus`. — [FRS #15 — Permissions](http://localhost:8080/root/trade-finance/-/issues/15#9-permissions)
- Target `ChecklistItem` exists and is not soft-deleted. — [FRS #15 — Commands](http://localhost:8080/root/trade-finance/-/issues/15#4-commands)
- `ConcurrencyStamp` matches the current stored value; mismatch throws `AbpDbConcurrencyException` (HTTP 409). — [FRS #16 — Concurrency](http://localhost:8080/root/trade-finance/-/issues/16#5-concurrency)

**Postconditions:**
- `Status` is flipped: `Active → Inactive` or `Inactive → Active`. — [FRS #15 — Transitions](http://localhost:8080/root/trade-finance/-/issues/15#5-transitions)
- `ConcurrencyStamp` is regenerated by ABP on update.
- `LastModificationTime` and `LastModifierId` are updated by the base class.
- No-op toggle behavior (attempting to toggle to the current state) is addressed by [IdempotentEditAuditBehavior](http://localhost:8080/root/trade-finance/-/wikis/decisions/IdempotentEditAuditBehavior): the toggle always flips the state by design (Active → Inactive, Inactive → Active), so a "same-state" toggle is architecturally impossible via this single-toggle contract. The idempotency concern applies only if the client sends two rapid consecutive toggles and both carry the original `ConcurrencyStamp` — the second will be rejected by the optimistic concurrency check.
- The response carries the full updated `ChecklistItemDto`.

**Domain events raised:** none — no external consumer in this milestone; state change is contained within the aggregate.

**Side effects:** none beyond persistence and audit trail.
