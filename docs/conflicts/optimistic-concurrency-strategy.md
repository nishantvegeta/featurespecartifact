**Node type:** Conflict
**Conflict ID:** CF-W05
**Source:** [FRS #16 — Concurrency](http://localhost:8080/root/trade-finance/-/issues/16#5-concurrency)
**Source FRS:** #16
**Conflict type:** `missing_precondition`
**Blocking severity:** medium

**Description:**
FRS #16 requires optimistic concurrency protection on Edit, Toggle, Reorder, and Delete operations for `ChecklistItem` to prevent lost updates when multiple BankAdmins operate on the same item concurrently. The FRS does not specify the token mechanism. Three candidates are applicable in the ABP / EF Core / PostgreSQL stack: ABP's `IHasConcurrencyStamp` (string GUID, `HasConcurrencyToken()` in EF mapping), EF Core `RowVersion` (database-managed timestamp or `xmin` in PostgreSQL), and a custom application-layer ETag managed without ABP framework support. The choice affects API contract (clients must supply the token in mutating requests), database schema, and alignment with existing ABP patterns in the project.

**Affected categories:**
- `ChecklistItem` entity: whether it implements `IHasConcurrencyStamp`
- `UpdateChecklistItemDescriptionInput`, `ToggleChecklistItemStatusInput`, `ReorderChecklistItemInput`, `DeleteChecklistItemInput` DTOs: `ConcurrencyStamp` field presence
- `GetChecklistItemForEdit` query response: must include the token for clients
- EF Core entity configuration: `HasConcurrencyToken()` mapping

**Resolution question:**
Should `ChecklistItem` use ABP's `IHasConcurrencyStamp` (string GUID token, managed by ABP's `SaveChanges` interceptor, `HasConcurrencyToken()` in EF Core mapping), EF Core `RowVersion` (`byte[]` column, `IsRowVersion()` in EF mapping, database-managed on PostgreSQL via `xmin` system column), or a custom ETag header managed by the AppService layer?

**Suggested options:**
- **Option A — ABP `IHasConcurrencyStamp` (adopted default):** String GUID token regenerated by ABP on every update. Pros: ABP-idiomatic; framework-managed; database-agnostic; integrates with ABP's `DbUpdateConcurrencyException` → `AbpDbConcurrencyException` mapping. Cons: clients must supply the stamp in every mutating request, requiring a prior GET to obtain it.
- **Option B — EF Core `RowVersion`:** Database-managed `byte[]` timestamp. Pros: no application-layer management; server-side token regeneration transparent to the domain model. Cons: PostgreSQL requires mapping to `xmin` (system column) via EF Core raw mapping — non-standard ABP pattern; adds DB-vendor coupling; `byte[]` tokens are less friendly in JSON APIs.
- **Option C — Custom ETag header:** AppService generates and validates ETag from entity state. Pros: flexible. Cons: duplicates framework capability; non-standard; no ABP exception wrapping support.

**Default if unresolved:** Option A (`IHasConcurrencyStamp`) per the [OptimisticConcurrencyStrategy](http://localhost:8080/root/trade-finance/-/wikis/decisions/OptimisticConcurrencyStrategy) decision. This is the adopted default used throughout this Phase 7 synthesis.
